Git子模块

文章目录

【注意】本文最后更新于 2022-07-21 ，文中内容可能已过时，请谨慎使用。

程序界有一句话叫做"不要重复造轮子", 当你开发一个项目实现某些功能, 或者你的项目中想要新增某些小功能的时候, 可能网上早就有人实现并分享到github上了. 当你想要在你的主项目中添加别人的轮子实现某些功能的时候, Git的子模块功能可以很好的帮助你完成代码版本管理.
而对于一个’不求上进’的初级程序员, 学会如何在万千轮子中找到适合自己的, 做一个优秀的’调库侠’, 甚至都能获得一份不错的工作(狗头)

参考文献

Git官方中文文档–Git子模块

补充说明: 由于本人目前只是在使用hugo搭建个人博客时用到git子模块, 并未有其他深入使用机会, 此文只做一些简单记录, 更多详情请参考官方文档.

添加子模块

要添加子模块, 使用git submodule add命令. 比如我们使用hugo博客系统写博客, 添加主题的时候, 就可以使用子模块功能添加其他人开发的现成的主题模块, 参考本人另一篇文章hugo加github pages搭建个人博客完全发布模式.

1~$ hugo new site myblog
2~$ cd myblog
3myblog$ git init
4myblog$ git submodule add [theme-repo] themes/[maupassant]

在博客主项目myblog中直接使用git submodule add命令, 后面添加主题子模块仓库地址[theme-repo]以及主题路径themes/[maupassant], 路径可以省略, 默认上在当前目录下新建与子模块仓库同名目录.

经过该操作之后, 主项目目录下会多出一个.gitmodules隐藏文件, 文件中记录着子模块的本地路径以及远程仓库地址. 以下上我的博客项目子模块相关信息

 1xy@TB14G2:myblog$ cat .gitmodules 
 2[submodule "public"]
 3	path = public
 4	url = git@github.com:xy-ylr/xy-ylr.github.io.git
 5[submodule "themes/maupassant"]
 6	path = themes/maupassant
 7	url = git@github.com:xy-ylr/maupassant-hugo.git
 8[submodule "content"]
 9	path = content
10	url = git@github.com:xy-ylr/blog-markdown.git
11[submodule "themes/even"]
12	path = themes/even
13	url = git@github.com:olOwOlo/hugo-theme-even.git

克隆含有子模块的项目

要克隆含有子模块的项目, 还是使用git clone命令, 不过该命令只会下载主项目的内容, 以及子模块根目录, 但是子模块数据还需要通过额外的命令来下载.

 1xy@TB14G2:~$ git clone git@github.com:xy-ylr/blog-frame.git Documents/ttt
 2Cloning into 'Documents/ttt'...
 3remote: Enumerating objects: 8, done.
 4remote: Counting objects: 100% (8/8), done.
 5remote: Compressing objects: 100% (5/5), done.
 6remote: Total 8 (delta 0), reused 8 (delta 0), pack-reused 0
 7Receiving objects: 100% (8/8), done.
 8xy@TB14G2:~$ cd Documents/ttt/
 9xy@TB14G2:ttt$ cat .gitmodules 
10[submodule "public"]
11	path = public
12	url = git@github.com:xy-ylr/xy-ylr.github.io.git
13[submodule "themes/maupassant"]
14	path = themes/maupassant
15	url = git@github.com:xy-ylr/maupassant-hugo.git
16[submodule "content"]
17	path = content
18	url = git@github.com:xy-ylr/blog-markdown.git
19xy@TB14G2:ttt$ la themes/maupassant/
20xy@TB14G2:ttt$ git submodule init 
21Submodule 'content' (git@github.com:xy-ylr/blog-markdown.git) registered for path 'content'
22Submodule 'public' (git@github.com:xy-ylr/xy-ylr.github.io.git) registered for path 'public'
23Submodule 'themes/maupassant' (git@github.com:xy-ylr/maupassant-hugo.git) registered for path 'themes/maupassant'
24xy@TB14G2:ttt$ git submodule update 
25Cloning into '/home/xy/Documents/ttt/content'...
26Cloning into '/home/xy/Documents/ttt/public'...
27Cloning into '/home/xy/Documents/ttt/themes/maupassant'...
28Submodule path 'content': checked out 'c2924a6ee4366976fa0df9d607527b692e717c4b'
29Submodule path 'public': checked out '3ca4354f28a5c1d4dc6de7f6aebb847bf63a7dc6'
30Submodule path 'themes/maupassant': checked out '76e036093e905b60f6d53df161b3472d19394220'
31xy@TB14G2:ttt$ la themes/maupassant/
32archetypes   .git        layouts     preview.png   README.md  theme.toml
33exampleSite  .gitignore  LICENSE.md  README_EN.md  static

由上述操作过程可以看出, 运行git submodule init和git submodule update之后, 才将数据检出到本地子模块目录.

当然, 也可以使用git submodule update --init命令将上述的两条命令合并成一条命令, 如果子模块中还有嵌套其他子模块, 可以再添加--recursive选项.

或者在克隆的时候直接使用git clone --recurse-submodules, 这条命令可以在克隆的时候, 自动初始化并检出子模块数据, 包括嵌套子模块.

获取子模块更新

一般情况下, 使用子模块的最简单方式是, 我们主要开发的还是自己的主项目, 子模块只是作为一个第三方库引入, 不对其做任何修改, 只是偶尔获取来自子模块上游的bug修复等更新.

要获取子模块更新, 可以直接进入子模块目录, 然后使用git fetch加git merge来合并上游的更新, 或者使用git pull, 或者使用git rebase也都可以, 跟在普通仓库中操作一样. 也可以直接在主项目中使用git submodule update --remote [repo]来获取更新.

1xy@TB14G2:ttt$ git submodule update --remote 
2xy@TB14G2:ttt$ git submodule update --remote content/
3xy@TB14G2:ttt$ git config -f .gitmodules submodule.content.branch test

如果不指定具体子模块, 会更新项目中的所有子模块, 默认更新的数据来自子模块仓库的master分支, 如果想要跟踪其他分支, 可以使用git config命令来设置, 比如上述命令中, 意思就是跟踪content子模块的test分支, 并将该配置保存在.gitmodules文件中.

如果确实需要在子模块中进行二次开发, 其实也可以把当作普通的git仓库, 只需要进入子模块目录中, 就可以使用fetch, pull, push等命令完成操作, 参考官方中文文档Git-基础, Git-分支, 或者我的另两篇文章Git基础配置及使用, Git分支

获取主项目更新

获取主项目更新和普通仓库获取上游更新没有区别, 只是需要注意的是, 对于子模块只是拉取提交记录, 而不会自动合并到本地, 需要额外运行git submodule update命令, 具体操作参考本文的克隆含有子模块的项目部分.

推送更新到远程仓库

如果只是在主项目中开发, 相关操作和普通git仓库没有区别. 如果有对子模块进行来二次开发, 那需要先提交并推送子模块的更新, 然后再推送主项目更新. 要不然其他人拉取主项目更新的时候, 由于子模块的更改都还在本地仓库未上传, 他们运行git submodule update的时候, 会提示子模块中不包含某某提交, 获取该提交失败. 此时可以添加配置选项, 当主项目推送更新的时候, 检查子模块是否已经完成推送, 或者自动推送子模块更新.

1<!-- 主项目推送的时候, 检查子模块是否已经完成推送 -->
2git config push.recurseSubmodules check
3<!-- 主项目推送的时候, 自动完成子模块的推送工作 -->
4git config push.recurseSubmodules on-demand
