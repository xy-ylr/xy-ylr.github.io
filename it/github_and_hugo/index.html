使用Github pages和hugo搭建个人静态博客系统

文章目录

【注意】本文最后更新于 2022-07-11 ，文中内容可能已过时，请谨慎使用。

正经人谁写日记, 咱发博客(狗头). 说干就干, 利用hugo和github pages给自己搭建一个博客系统.

本文主要记录个人的搭建过程, 内容包括hugo博客本地化部署, 发布到github实现互联网访问, 接入第三方评论系统, 使用图床存储博客图片等

Hugo博客系统

官方介绍Hugo是由Go语言实现的静态网站生成器。简单、易用、高效、易扩展、快速部署。

确实是非常简单易用, 生成一个站点, 添加一个主题, 然后就可以开始使用Markdown格式写文章了, 一点都不极客(狗头)

其他类似的还有Jekyll和Hexo等, 不过貌似还是Hugo最快.

Hugo下载安装

本人操作系统ubuntu22.04, 采用snap安装:

1xy@TB14G2:~$ snap install hugo --channel=extended
2xy@TB14G2:~$ hugo version

基础安装不需要添加参数, 直接运行snap install hugo即可.

添加参数--channel=extended可以获取Sass/SCSS支持, 有些高大上的主题会需要这玩意.

两种版本可以通过snap refresh hugo --channel=extended 或者snap refresh hugo --channel=stable来转换.

如果后面输入hugo version能够看到版本号, 那就说明安装成功了.

其他操作系统及安装方式参考官方文档hugo install

本地生成一个博客站点

本地生成一个博客站点也只需要一条命令即可:

1xy@TB14G2:~$ hugo new site blog
2Congratulations! Your new Hugo site is created in /home/xy/blog.
3...
4

这样就表示创建成功了, 博客站点名称为blog, 里面就是该站点相关的各种文件

Hugo博客目录结构

新生成的博客站点目录结构如下

.
├── archetypes          // 使用hugo new 新建文章时候的文章模板
├── config.toml         // hugo站点的配置文件, 都在这里面了, 可以拷贝主题里的配置文件
├── content             // 写博客时候的markdown笔记, 都放在这里面
├── data                // 基础使用好像不用关注
├── layouts             // 各种html模板, 用于渲染你的MD笔记,不用关注,基础使用的话主题里已经包含这部分内容了
├── static              // 放js,css文件的, 不用关注,主题里也有了
├── public              // 默认不存在,运行`hugo`命令后出现, 用于存放生成的静态站点资源.[重点]
└── themes              // 博客主题目录, 刚生成的时候也是一个空目录, 相关介绍见后文
    ├── archetypes      // 将该目录中的default.md复制到根目录的archetypes就好了,或者自己自定义一个
    ├── layouts         // 用于渲染MD笔记的各种html模板,
    ├── static          // 放js/css文件的
    ├── exampleSite     // 主题自带的示例文章等, config.toml(后缀名也可能是yaml或者json)拿来用, 其他可删除
    └── [others]        // 其他各种文件/目录, 用于实现各种功能, 基本不用关注

其他更多内容同样参考 官方文档Directory Structure

添加一个博客主题

主题是用来渲染你的MD文章的, 你只需要写最简单的内容, 然后主题帮你展示成酷炫的页面. 网上现成的hugo主题有很多, 可以去 Hugo官方主题 找喜欢的博客主题, 或者去 Github hugo主题找也可以, 都是可以预览的, 挑一个喜欢的主题, 拿来即用.

我用的是 飞雪无情 的maupassant-hugo主题, 比较简洁. 点击链接跳转到页面下载ZIP包. 参考上面 hugo博客目录结构 进行下列操作:

  1. 下载的压缩包解压到blog/theme目录下, [可选]重命名主题目录.
  2. 将exampleSite/config.toml复制到站点根目录替换原有的配置文件, [如果重命名了主题目录, 注意修改文件中的theme选项].
  3. 将archetypes/defalut.md文件复制替换到站点根目录同名文件.
  4. [可选]将exampleSite/content中的示例文章复制到站点根目录的content目录下.
  5. 阅读README.md说明文档以及config.toml中的注释信息, 为后续其他配置调整做准备.

经过上述步骤就算完成博客主题的添加了, 其中第4步可以等后面根据自身需求慢慢调整.

尝试写一篇博客并预览效果

写博客可以直接在站点根目录/content/posts目录中新建.md文件, 但是推荐使用hugo命令来新建:

1xy@TB14G2:~$ hugo new posts/first_blog.md
2Content "/home/xy/Documents/blog/content/posts/first_blog.md" created
3xy@TB14G2:~$ hugo server -D

上述命令的意思是: 在站点根目录blog下/content/posts目录下新建文章first_blog.md, 使用该命令新建的文件会包含一些基本信息, 这些基本信息来自archetypes/default.md.

 1+++
 2title='{{ replace .Name "-" " " | title }}'
 3tags=[]
 4categories=[]
 5keywords=""
 6description=""
 7date="{{ .Date }}"
 8toc=true
 9draft=true
10hiddenFromHomePage= false
11+++
12
13<!--more-->
14
15

视你的default.md文件而定, 可能跟我的内容不尽相同. 其中draft=true表示该篇文章目前是草稿, 等到正式发布的时候, 需要修改为draft=false; toc选项是单独控制文章是否启用目录导航; hiddenFromHomePage选项用于控制是否在站点首页文章列表显示该篇文章.

创建文章后随便写点东西, 然后运行的hugo server -D作用是预览博客站点, 其中的参数-D的意思是预览草稿.

然后根据提示, 在浏览器中输入http://localhost:1313/ 就能看到网站的效果了, 可以使用组合按键Ctrl+C来结束预览.

发布博客到Github

经过以上操作, 你已经成功的在本地搭建起了一个hugo博客系统, 并可以愉快的开始写博客了. 但是富贵不还乡, 犹如锦衣夜行, 既然写了博客, 自然好文共赏析, 要发出去让让广大网友点评一二^_^

正好, Github pages支持发布静态站点, 而且Github可以免费注册使用, 所以, 赶紧白嫖吧~~. 国内的gitee也可以, 而且速度更快, 只是github面向全球, 显得更主流一些(github偶尔可能国内会无法访问,或者访问很慢). 另外就是gitee前阵子刚进行了一些调整, 经过简单尝试后发现目前大概有以下几个小问题(描述不一定准确, 影响也不算大, 看个人取舍吧):

  • 需要登录后才能浏览仓库,
  • 创建公共仓库gitee pages需要实名认证

Git基础

用于跟github交互的版本控制工具git的下载安装以及基础使用等, 请参考官方文档, 或者参考我的另一篇博客, 或者自行网上搜索.

创建github仓库

首先需要注册一个github帐号, 然后新建一个仓库用于发布博客, 选择public模式, 这样所有人都可以通过互联网看到你的博客.

注意仓库名, 如果仓库名是Owner.github.io格式的, 可以直接通过仓库名来访问, 否则需要使用https://owner/github.com/repo的格式来访问仓库, 这种二级目录访问格式可能会让部分链接解析出错, 所以推荐使用前者.

仓库创建完成后, 点击页面上的Settings, 然后选择侧边栏的Code and automation -> pages, 开通github pages功能.

精简发布模式

在博客根目录下运行hugo命令, 会在/public/目录下生成一系列的文件, 包含了一个静态站点所需的所有内容, 所以只需要把/public目录下的内容上传到github就可以了. 首先命令行进入public目录, 然后使用以下命令

 1// 初始化本地仓库
 2$ git init
 3// 添加远程仓库, 推荐使用ssh模式,下面的owner替换成你自己的实际仓库名, 也可以使用https模式
 4$ git remote add git@github.com:owner/owner.github.io.git
 5// 暂存本地仓库文件
 6$ git add .
 7// 提交更改到本地仓库
 8$ git commit -m "第一次发布博客测试"
 9// 将本地仓库内容基于远程仓库进行重做
10$ git rebase origin/master
11// 推送本地仓库内容到远程仓库
12$ git push origin

远程仓库可能存在本地仓库没有的readme.md等初始化文件, 使用git rebase进行重做而不是git merge合并, 是rebase更加线性. 更多关于两条命令的区别参考官网

当然, 也可以在rebase之后先检查一下public目录, 如果有多出来的无用文件, 可以使用git rm 先删除, 然后重新git commit, 最后再push到远程仓库.

到这个时候, 第一篇博客就已经成功发布出去了, 过一会儿, 就可以通过owner.github.io查看结果了.

完全发布模式(git子模块)

尝试过精简发布模式之后, 我们就会发现, 我们只推送了public目录里的静态站点资源, 其他的比如hugo主框架, theme目录下的主题, content目录下的MD源文件这些东西都在本地没有上传.

在极端环境下, 比如电脑因为意外损坏了, 或者被人永久删除了博客目录, 这样就会导致public之外的其他数据丢失. 又或者有多台电脑同时在写博客, 这也需要至少保持content目录下的MD文件数据同步. 所以我们需要整个博客目录的数据都有一个备份, 方便必要的时候能够快速恢复博客系统环境. 但是, 把整个博客系统都放在一个仓库里发布出去, 也没有必要; 主题也是用的别人的略加修改而来的, 如果能保持一个独立的仓库, 也更容易应用别人的更新…

综合考虑, 我们可以使用git子模块功能, 将整个博客系统拆分成如下四个部分, 相对应的也需要四个远程仓库:

功能模块 模块说明 仓库名(参考) 仓库说明
博客主项目 hugo博客系统主框架 blog-main private,
主题子模块 themes博客使用的主题 [some-theme] public, 从你喜欢的主题仓库那fork, 保持同名即可,
如果有做了一些有趣的修改, 也可供其他人下载使用.
笔记子模块 content目录下的markdown源文件 blog-md private, 笔记源文件无需公开
站点子模块 public目录下的静态站点资源 owner.github.io public, 对外展示供他人浏览的博客仓库

以下是大概过程(可能有部分错漏, 欢迎评论区指正) 注意$提示符前面的目录变化, 注意[]部分的内容替换成你自己的:

 1// 新建博客站点
 2~$ hugo new site blog
 3// 初始化为git仓库
 4~$ cd blog/
 5blog$ git init
 6// 添加远程仓库主框架
 7blog$ git remote add main [blog-main-repo]
 8// 添加子模块仓库, 其中/content和/public目录已经初始化到主仓库中, 需要先删除
 9blog$ rm -r content/ public/
10blog$ git submodule add [theme-repo] themes/[maupassant]
11blog$ git submodule add [blog-md-repo] content/
12blog$ git submodule add [owner.github.io] public/
13// 修改主题仓库部分配置, 用于适配本地博客系统, 然后将更改后的主题重新推送回远程仓库
14blog$ cd themes/maupassant
15maupassant$ git add .
16maupassant$ git commit -m "备注本次提交内容概要"
17maupassant$ git push origin
18// 写博客, 预览效果, 确认完成后推送笔记到远程笔记仓库
19maupassant$ cd ../../
20blog$ hugo new posts/my_first_blog.md
21blog$ hugo server -D
22blog$ cd content/
23content$ git add .
24content$ git commit -m "我的第一篇博客"
25content$ git push origin
26// 生成静态资源, 然后推送到远程仓库
27content$ cd ../
28blog$ hugo
29blog$ cd pubic/
30public$ git add .
31public$ git commit -m "我的第一篇博客"
32public$ git push origin
33// 推送博客主框架到远程仓库, 
34public$ cd ../
35blog$ git add .
36blog$ git commit -m "博客主框架已初步构建完成"
37blog$ git push main
38

后续主题部分可能就不再有变化了, 那就不用再推送. 而content笔记部分和public静态资源部分, 会随着我们不停的写博客而变化.

主框架也会随着content和public两个子模块的变化而变化, 不过这个可以忽略不管, 毕竟主框架就真的只是一个框架, 东西都在其他几个子模块上.

其他配置或改动

接入utteranc评论系统

utteranc评论系统需要登录github才能评论, 而github偶尔可能会无法访问, 或者访问很慢, 会影响体验. 本来想改用waline评论系统, 不需要登录, 只要设置昵称/邮箱就可以评论. 但是waline又需要注册LeanCloud数据库(国内版需要绑定已备案域名, 国际版同样存在速度慢), 再使用Vercel 或者阿里云函数计算等部署 (服务端)也是很麻烦. 所以暂时还是使用utteranc, 以后看情况再更改评论系统.

接入utteranc评论系统很简单, 参考官方说明文档, 其实只需要创建一个github公共仓库, 访问utterances app将应用接入该公共仓库就已经完成了. 然后将该仓库对应的owner/repo复制粘贴到config.toml配置文件相应位置就算接入成功了.

如果你选择的主题不支持接入该评论系统(使用别的如valine, waline等评论系统), 而你又想用utteranc, 那会稍微麻烦一点点, 大概在主题目录的layouts/partials目录下应该有一个模板文件comments.html, 将utterances的脚本代码复制进去参考着其他评论系统稍作修改也就可以了.

1<script src="https://utteranc.es/client.js"
2        repo="[ENTER REPO HERE]"  /* 你的owner/repo */
3        issue-term="pathname"
4        theme="github-light"
5        crossorigin="anonymous"
6        async>
7</script>

使用图床存储图片

使用github pages来托管博客, 由于国内访问github速度比较慢. 而图片,特别是高清图片一般都比较大, 妥妥的吃流量大户, 会更进一步拖慢博客页面打开速度, 所以将博客中用到的图片存储在图床上, 再通过外链来引用.

国内的免费图床有很多, 比如七牛云, 开通七牛云对象存储服务的实名认证用户, 拥有10GB的免费存储空间, 10GB/月的免费CDN回源流量等, 正好拿来做图床用, 只要不超标, 那就是免费的, 更多七牛云对象存储计费方式, 不过用七牛云图床需要一个已备案的域名, 这个是要钱的.

又比如聚合图床, 免费注册, 免费版有1万张图片可以放, 如果就只是放博客图片的话, 那也能写好多好多年博客了, 哈哈哈.

主题自定义菜单纳入文章归档

原主题只归档content目录下post, posts目录的文章, 但是我将博客文章分了’计算机’, ‘健身’等几个大分类, 并分别创建独立目录(未将所有文章都放在posts目录下), 结果我新建的几大分类的文章在博客首页/归档页都不会出现, 以及右侧边栏的’最近文章’也不会出现, 所以对部分模板页面做了一些微调.

修改的位置是主题目录下的layouts/archives/single.html, layouts/partials/recent_post.html和layouts/index.html, 将it, fitness等分类目录名加入数组:

1    {{ range (where (where .Site.Pages "Type" "in" (slice "post" "posts" "it" "fitness")) "Kind" "page").GroupByDate "2006" }}

还有一个版权申明模板页面layouts/partials/copyright.html, 默认也是只有post或者posts目录下的文章会有版权申明, 也需要修改.

说明: 其实这个改动没有太大必要, 本身主题就是支持分类的, 只要给文章设定好分类, 相应的文章就自动实现聚合浏览了. 为只是为了将几个大类分开, 并在自定义菜单上显示出来能更醒目一点. 也许这样子用上一段时间后, 又要把这个改动调整回去了…

主题文章目录长度溢出自动换行

原主题文章只显示一级目录, 二级目录被隐藏了, 而且目录文字过长不会自动换行, 修改相应的css

修改位置: layouts/partials/toc.html, 注释了2行css, 其他未改动.

 4        /* white-space: nowrap; */
 5        ...
 6    }
 7    .post-toc .post-toc-content ul ul {
 8        /* 显示二级目录 */
 9        /* display: none; */
10        ...
11    }
12</style>

代码块无法对齐

代码块部分的渲染效果, 原主题左边的行号和右边的代码无法对齐. 该主题使用的是hugo原生的代码高亮方式, 阅读官方文档 - 代码高亮配置, 经过调试发现, 只要调整lineNumbersInTable参数即可, 修改后config.toml相关配置:

1## 配置代码高亮
2[markup]
3  [markup.highlight]
4    lineNos = true
5    lineNumbersInTable = false      ## 默认为true,行号与代码块相互独立, 导致行号与代码块无法对齐
6    style = 'solarized-dark'
7    tabWidth = 4

其他样式微调

1, 在主题样式表static/css/sytle.css中, 对于列表的控制是

1.post-content ul {
2    overflow: auto;
3    padding: .5em 2.4em;
4    border-radius: 3px;
5    margin:1.8em 0; /* 会影响嵌套列表 */
6}
7.post-content ul li{
8    text-indent: .5em;
9}

当文章中有嵌套列表的时候, 会导致中间有很大的空白, 额外新增嵌套列表样式, 去掉多余的外边框

1.post-content ul ul{
2    margin: 0;
3}

添加文章预计阅读时间

参考even主题, config.toml配置文件[params]选项下添加参数moreMeta = true, 相关页面(single.html)添加如下代码

1        {{- if .Site.Params.moreMeta }}
2          <span class="more-meta"> {{ T "wordCount" .WordCount }} </span>
3          <span class="more-meta"> {{ T "readingTime" .ReadingTime }} </span>
4        {{- end }}

直接照搬上述代码无法生效, 这是由于两个主题的模板解析语法略有差异, even主题是{{- if xxx }}xxx{{- end }}构造, 而maupassant主题是{{ if xxx }}xxx{{ end }}构造.

查阅hugo官方文档后, 确认WordCount, ReadingTime两个变量是hugo原生支持, 点击查看相关介绍. 而moreMeta是配置文件中自定义的一个参数, 用于控制WordCount和ReadingTime两个变量. 而{{ T "wordCount" .WordCount }}是因为even主题包含多语言功能, 为了实现相关内容的自动翻译, 将文本内容都放在来多语言模板里, 比如简体中文语言模板/i18n/zh-CN.yaml中的部分内容:

wordCount:
  one: "约 {{ .Count }} 字"
  other: "约 {{ .Count }} 字"

readingTime:
  one: "预计阅读 {{ .Count }} 分钟"
  other: "预计阅读 {{ .Count }} 分钟"

我们不需要多语言支持, 直接写成下面这样就可以了, 不过统计的字数可能不准, 譬如本文, Typora编辑器帮我统计的字数是5300出头, 而hugo为我统计的字数已经远超7000了(捂脸)

1        {{if .Site.Params.moreMeta }}
2        <div class="post-meta">
3            <span>|</span>
4            <span> 共约{{ .WordCount }}字 </span>
5            <span> 预计阅读时间{{ .ReadingTime }}分钟</span>
6        </div>
7        {{ end }}
