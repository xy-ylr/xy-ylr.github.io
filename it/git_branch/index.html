Git分支

【注意】本文最后更新于 2022-07-21 ，文中内容可能已过时，请谨慎使用。

在实际项目开发中, 可能会存在稳定版, 先行版, 内部开发版等, 又或者正在开发新版功能的时候, 老版本爆出致命漏洞需要修复, 又或者开发功能1的时候, 出现了新需求需要满足(功能2), 诸如此类, Git的分支功能可以很好的解决上述问题

分支简介

具体内容参考官方Git分支简介, 本人能看明白, 但是无法举一反三深入理解, 也就记不住, 更加无法自我组织言语表达出来.

简单理解, Git仓库系统由三个部分组成,

  • blob对象: 保存文件快照, 每个文件一个快照
  • 树对象: 记录仓库目录结构和blob对象的索引
  • 提交对象: 记录指向树对象的指针和提交信息

而Git分支, 是额外的指向提交对象的可移动指针. 比如Git默认创建的master分支, 在默认单分支开发状态下, 不管经过多少次git commit, 它总是指向最新的一次提交. 可以使用git log 来观察这一现象

1xy@TB14G2:subtest$ git log --oneline --graph 
2* 6fd8734 (HEAD -> master, origin/master, origin/HEAD) save the blog article tmp
3* c52cf19 modify hhh.html
4* 2b73bb7 add test.md
5* 9035ec4 modify git小问题临时记录.md
6* 4e0029d add active
7* 1da5508 Initial commit

其中的HEAD -> master是本地仓库的master分支, 而HEAD是一个特殊指针, 表示当前(工作目录)所处的分支. 而origin/master, origin/HEAD都表示的远程仓库分支.

了解这些基本概念后, 开始分支练习吧.

创建新分支

要创建分支使用git branch命令, 这会在当前提交对象上新建一个指针.

1xy@TB14G2:subtest$ git branch test

查看分支

上述新建分支成功后不会有什么输出反馈, 可以通过以下命令来查看并验证结果

1xy@TB14G2:subtest$ git branch -vv
2* master 6fd8734 [origin/master] save the blog article tmp
3  test   6fd8734 save the blog article tmp
4xy@TB14G2:subtest$ git log --oneline --decorate
56fd8734 (HEAD -> master, origin/master, origin/HEAD, test) save the blog article tmp
6c52cf19 modify hhh.html

上述两条命令, git branch -vv命令是列出所有本地分支基本信息以及远程分支名称, 本地分支与远程分支的基本差异, 也可以直接使用git branch, 那就只显示分支名; 或者使用git branch -v, 与-vv相比, 少了远程分支名称. 其输出结果里的 *类似于git log里的HEAD, 都是表示当前活动分支的意思.

切换分支

通过刚才的验证结果可以发现, test分支和master分支都指向当前最新提交记录(6fd8734), 但是当前活动分支还是master, 也就是HEAD指针还指向master.

要想将当前活动分支也就是HEAD指针切换到test分支, 可以使用命令git checkout [branch]

1xy@TB14G2:subtest$ git checkout test 
2Switched to branch 'test'
3xy@TB14G2:subtest$ git branch -vv
4  master 6fd8734 [origin/master] save the blog article tmp
5* test   6fd8734 save the blog article tmp

可以看到, 活动分支确实已经切换到test了. 另外, 也可以使用git checkout -b创建新分支并同时切换过去.

接下来我们在两个分支上分别进行一些改动

 1// 在test分支新增文件并提交
 2xy@TB14G2:subtest$ echo "ttttt" > test.md
 3xy@TB14G2:subtest$ git add .
 4xy@TB14G2:subtest$ git commit -m "branch testing"
 5[test f410fd9] branch testing
 6 1 file changed, 1 insertion(+)
 7 create mode 100644 test.md
 8// 切换会master分支并做修改提交
 9xy@TB14G2:subtest$ git checkout master 
10Switched to branch 'master'
11Your branch is up to date with 'origin/master'.
12xy@TB14G2:subtest$ echo "mmmmm" > test.html
13xy@TB14G2:subtest$ git add .
14xy@TB14G2:subtest$ git commit -m "add test.html"
15[master 8e118a5] add test.html
16 1 file changed, 1 insertion(+)
17 create mode 100644 test.html
18xy@TB14G2:subtest$ 
19xy@TB14G2:subtest$ git log --oneline --decorate --graph --all
20* 8e118a5 (HEAD -> master) add test.html
21| * f410fd9 (test) branch testing
22|/  
23* 6fd8734 (origin/master, origin/HEAD) save the blog article tmp
24* c52cf19 modify hhh.html

能够看到, test分支已经向前一步指向新的提交对象(f410fd9), 而master分支也指向了一个新的提交对象(8e118a5), 两个分支分叉了.

合并分支

现在假设master分支是你正经开发的东西, test分支是一个突发奇想, 实现之后发现确实很妙, 准备把它整到master主分支中来, 这可以使用Git的分支合并功能来实现.

 1xy@TB14G2:subtest$ git merge test 
 2Merge made by the 'ort' strategy.
 3 test.md | 1 +
 4 1 file changed, 1 insertion(+)
 5 create mode 100644 test.md
 6xy@TB14G2:subtest$ 
 7xy@TB14G2:subtest$ git log --oneline --decorate --graph --all
 8*   4df411c (HEAD -> master) Merge branch 'test'
 9|\  
10| * f410fd9 (test) branch testing
11* | 8e118a5 add test.html
12|/  
13* 6fd8734 (origin/master, origin/HEAD) save the blog article tmp
14* c52cf19 modify hhh.html

可以看到, master分支从原来的8e118a5变成了4df411c, 而且这个新提交记录是由f410fd9(test)跟8e118a5(就master)合并而来. 这是因为原本master和test就分叉了, 所以Git在合并的时候基于他们的共同历史提交记录, 进行了一次三方整合, 然后自动生成一个新的提交记录.

相应的, 也有一个更简单的合并模式, 就是master和test处在同一条提交记录线上, 只是master比test的记录老, 那在合并的时候, 就只会把master的指针移动到test所指的记录上, 而不会生成新的提交记录.

冲突合并

另外值得注意的是, 由于是分叉的两个分支进行合并, 那么就有可能某些文件在两个分支进行了不同的修改, 在这种情况下, Git会中止合并, 然后将有冲突的文件都列出来, 等待我们手动去除冲突, 并使用git add命令将冲突文件重新加入暂存区后, 此次合并才算完成.

 1// 在master分支编辑并提交test.md
 2xy@TB14G2:subtest$ echo "master branch mmmmm" >> test.md 
 3xy@TB14G2:subtest$ git commit -am "master changes"
 4[master 8e14975] master changes
 5 1 file changed, 1 insertion(+)
 6// 退回4df411c位置新建分支ttest并切换过去, 然后编辑并提交test.md
 7xy@TB14G2:subtest$ git checkout -b ttest 4df411c
 8Switched to a new branch 'ttest'
 9xy@TB14G2:subtest$ echo "ttest branch bbbbb" >> test.md 
10xy@TB14G2:subtest$ git commit -am "ttest change"
11[ttest ee77788] ttest change
12 1 file changed, 1 insertion(+)
13xy@TB14G2:subtest$ git log --oneline --graph --decorate --all
14* ee77788 (HEAD -> ttest) ttest change
15| * 8e14975 (master) master changes
16|/  
17*   4df411c Merge branch 'test'
18|\  
19| * f410fd9 branch testing
20* | 8e118a5 add test.html
21|/  
22* 6fd8734

可以看到, master分支与ttest分支也分叉了, 接下来我们再次尝试merge合并分支.

 1xy@TB14G2:subtest$ git checkout master 
 2Switched to branch 'master'
 3Your branch is ahead of 'origin/master' by 4 commits.
 4  (use "git push" to publish your local commits)
 5xy@TB14G2:subtest$ git merge ttest 
 6Auto-merging test.md
 7CONFLICT (content): Merge conflict in test.md
 8Automatic merge failed; fix conflicts and then commit the result.
 9xy@TB14G2:subtest$ git status 
10On branch master
11Your branch is ahead of 'origin/master' by 4 commits.
12  (use "git push" to publish your local commits)
13
14You have unmerged paths.
15  (fix conflicts and run "git commit")
16  (use "git merge --abort" to abort the merge)
17
18Unmerged paths:
19  (use "git add <file>..." to mark resolution)
20	both modified:   test.md
21
22no changes added to commit (use "git add" and/or "git commit -a")
23xy@TB14G2:subtest$ vi test.md

合并结果的最后一行提示: 自动合并失败, 修复冲突然后提交结果. 然后运行git status也有类似提示, 并说明冲突文件是test.md. 如果此时我们再运行git log --oneline --graph --decorate --all命令, 也可以看到提交历史还没有发生变化.

接下来我们打开test.md文件进行修复, 内容如下

1ttttt
2<<<<<<< HEAD
3master branch mmmmm
4=======
5ttest branch bbbbb
6>>>>>>> ttest

Git使用了<<<<<<<, =======, >>>>>>>等来标记冲突内容, 其中=======上面的部分是HEAD也就是master分支的内容, 而下面的部分是test分支的内容.

 1xy@TB14G2:subtest$ cat test.md 
 2ttttt
 3master branch mmmmm
 4ttest branch bbbbb
 5xy@TB14G2:subtest$ git commit -am "fix conflicts"
 6[master b11a866] fix conflicts
 7xy@TB14G2:subtest$ 
 8xy@TB14G2:subtest$ git log --oneline --graph --decorate --all
 9*   b11a866 (HEAD -> master) fix conflicts
10|\  
11| * ee77788 (ttest) ttest change
12* | 8e14975 master changes

我们根据实际需求进行修改, 修改完成后将这些特殊标记符都去掉, 然后重新提交, 可以看到新的提交记录是由ee77788和8e14975合并而来.

变基

通过上述分支合并测试操作练习, 随着项目的发展, 分叉状态的分支合并操作会越来越多, 到后面有可能出现一个类似网状的历史提交记录, 到那个时候, 对着一张乱七八糟的历史提交记录, 免不了会头痛眼晕. 所以Git提供了另一种操作方式–变基

变基的效果是: 基于最近的共同提交对象, 将需要合并的分支上的修改内容提取出来, 然后在目标分支上重新来一遍, 使得分叉的分支合并变成线性推进.

更多关于变基的操作, 变基的风险, 以及变基与合并的差异, 请参考官方文档-变基, 此处仅演示其中一个例子.

 1xy@TB14G2:subtest$ git checkout -b rb
 2Switched to a new branch 'rb'
 3xy@TB14G2:subtest$ la
 4.git  post  README.en.md  README.md  test.html  test.md
 5xy@TB14G2:subtest$ echo "git rebase test" > rebase.md
 6xy@TB14G2:subtest$ git add .
 7xy@TB14G2:subtest$ git commit -m "rb test"
 8[rb af4ec31] rb test
 9 1 file changed, 1 insertion(+)
10 create mode 100644 rebase.md
11 xy@TB14G2:subtest$ git checkout master
12 Switched to branch 'master'
13xy@TB14G2:subtest$ echo "rebase test" > rebase.md
14xy@TB14G2:subtest$ git add .
15xy@TB14G2:subtest$ git commit -m "rebase test"
16[master dbd91b5] rebase test
17 1 file changed, 1 insertion(+)
18 create mode 100644 rebase.md
19xy@TB14G2:subtest$ git log --oneline --decorate --graph --all
20* af4ec31 (rb) rb test
21| * dbd91b5 (HEAD -> master) rebase test
22|/  
23*   b11a866 fix conflicts
24...

经过上述过程, 分别在master分支和rb分支编辑了文件rebase.md, 两个分支再次分叉, 现在使用git rebase命令开始变基操作

1xy@TB14G2:subtest$ git rebase master rb 
2Auto-merging rebase.md
3CONFLICT (add/add): Merge conflict in rebase.md
4error: could not apply af4ec31... rb test
5hint: Resolve all conflicts manually, mark them as resolved with
6hint: "git add/rm <conflicted_files>", then run "git rebase --continue".
7hint: You can instead skip this commit: run "git rebase --skip".
8hint: To abort and get back to the state before "git rebase", run "git rebase --abort".
9Could not apply af4ec31... rb test

由于两个分支变基的都是rebase.md文件,所以出错,需要手动修复.此处可以运行git status查看提示信息

 1xy@TB14G2:subtest$ git status 
 2interactive rebase in progress; onto dbd91b5
 3Last command done (1 command done):
 4   pick af4ec31 rb test
 5No commands remaining.
 6You are currently rebasing branch 'rb' on 'dbd91b5'.
 7  (fix conflicts and then run "git rebase --continue")
 8  (use "git rebase --skip" to skip this patch)
 9  (use "git rebase --abort" to check out the original branch)
10
11Unmerged paths:
12  (use "git restore --staged <file>..." to unstage)
13  (use "git add <file>..." to mark resolution)
14	both added:      rebase.md
15
16no changes added to commit (use "git add" and/or "git commit -a")

接下来我们手动修复rebase.md, 然后重新提交.

1xy@TB14G2:subtest$ vi rebase.md
2xy@TB14G2:subtest$ git add .
3xy@TB14G2:subtest$ git commit -m "fix rebase"
4[detached HEAD 9350efc] fix rebase
5 1 file changed, 1 insertion(+), 1 deletion(-)

最后根据之前的提示运行命令git rebase --continue

1xy@TB14G2:subtest$ git rebase --continue 
2Successfully rebased and updated refs/heads/rb.
3xy@TB14G2:subtest$ git log --oneline --decorate --graph --all
4* 9350efc (HEAD -> rb) fix rebase
5* dbd91b5 (master) rebase test
6*   b11a866 fix conflicts

对比变基之前的提交历史, 本来rb分支跟master分支是分叉的, 但是现在分叉消失了, 而且rb分支从原来的af4ec31变成了现在的9350efc.

删除分支

上述测试中, test分支, 以及ttest分支的内容都合并到master主分支中了, 已经不需要这两个分支了, 那么我们就可以删除掉这两个分支.

1git branch -d ttest
2git branch -D ttest
3git branch --merged
4git branch --no-merged

如果要删除的分支包含未被合并的内容, 需要使用-D选项, 删除之后, 未被合并的内容也会一并被删除. 可以使用--merged 或者--nomerged来查看分支(相对于当前活动分支)合并情况.

远程分支

如果要将本地的(master主分支以外)某个分支比如test分支, 也推送到远程仓库, 可以使用

1git push origin test
2git push origin test:otherbranch

第一条命令是将本地test分支推送到origin远程仓库的test分支, 第二条命令是将本地test分支推送到origin远程仓库的otherbranch分支.

如果要获取远程仓库默认分支以外的其他分支的内容, 先使用git fetch origin将origin远程仓库信息同步到最新状态, 然后使用如下命令, 意思是本地新建test分支, 用于检出origin远程仓库的test分支的数据.

1git checkout -b test origin/test

如果要为当前分支设置跟踪远程仓库的某个分支, 可以使用如下命令

1git branch -u origin/test

要查看远程分支的相关信息, 可以使用git remote show <remote>, 或者git branch -vv命令

相关命令汇总

命令 说明
git branch 查看分支列表
git branch test 新建分支并命名为test
git branch -v 查看各分支最后一次提交, 各分支与远程分支的同步差异
git branch -vv 查看各分支最后一次提交, 各分支与远程分支的同步差异, 以及远程分支名称
git checkout test 将当前活动分支切换为test分支
git checkout -b test 新建test分支, 并切换过去
git merge test 将当前活动分支与test合并分支
git rebase master test 将test分支的内容提取出来基于master分支重做
git rebase --onto master server client 将client分支的内容基于server分支提取出来, 然后应用到master分支上
git branch -u origin/test 设置本地当前分支关联origin远程仓库的test分支
