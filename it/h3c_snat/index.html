要设置源地址nat，如果只是少量地址需求，可以使用静态地址转换技术，这是一种内外网IP地址的转换关系固定的NAT方式，最适用于少量地址的一对一转换；如果是大量用户都需要nat，那可以使用动态nat技术，用于将大量内网地址转换为少量外网地址。

模拟环境网络拓扑大致如下图

前期环境准备，端口、路由可达、防火墙安全域配置等，以及其他nat配置、各nat区别总结，请看博客文章h3c防火墙nat配置。

少量用户nat访问外网

关键命令：nat static outbound local_ip global_ip

local_ip是指内部想要访问外部网络的IP地址，global_ip是指nat转换后的外网IP地址。

假设现在要让nw1以静态转换的方式访问ww，转换后的地址是10.10.10.254，当然，也可以直接用出接口地址10.10.10.2。

我们可以在nw1交换机上telnet登录ww看看结果，可以发现在ww交换机上显示的登录用户IP是10.10.10.254。而之后我们再从nwhx交换机telnet过去，显示的登录用户就直接是8.8.8.2。这也说明我们的NAT配置成功了。

同时，通过nat地址来反向telnet也是可以的，如下图所示，在ww上运行telnet 10.10.10.254，最终登录到了nw1上。

添加acl限制

只是这么配置的话，ww与nw1是一个全端口放行状态，但实际可能ww只是有个web服务提供给nw1访问，其他业务并不对nw1开放，这个时候我们就需要进一步添加acl来限制访问范围了。

（当然，实际上，遇到这种情况，更可能是在ww侧的边界防火墙做nat以及限制等，毕竟权限得握在自己手里才更放心。）

1
2
3
4
5
6
7
8

# 先删除刚才的nat static配置
[fw]undo nat static outbound 1.1.1.11 10.10.10.254
# 配置acl策略，只允许1.1.1.11访问10.10.10.1的80端口
[fw]acl advanced 3000
[fw-acl-ipv4-adv-3000]rule permit tcp source 1.1.1.11 0 destination 10.10.10.1 0 destination-port eq 80
[fw-acl-ipv4-adv-3000]qu
# 配置带acl限制的nat static转换
[fw]nat static outbound 1.1.1.11 10.10.10.254 acl 3000

此时我们再次从nw1发起telnet，可以发现在ww上显示的登录用户IP是1.1.1.11，虽然登录成功了，但这只是由于两边路由相通，且前期防火墙配置中放行了两个安全域之间的telnet服务，NAT转换实际上并没有进行。

而如果我们再添加一条策略，放行telnet服务端口

1
2

[fw]acl advanced 3000
[fw-acl-ipv4-adv-3000]rule permit tcp source 1.1.1.11 0 destination 10.10.10.1 0 destination-port eq 23

那么telnet将再次经过nat转换并登录成功。但是由于acl策略的限制，此时反向telnet还是无法成功，如下图所示。

当然，这只是为了说明带acl限制的静态转换，实际上用处并不大，因为结合前期防火墙配置，我们只是限制了1.1.1.11到10.10.10.1的流量需要nat，而并没有限制nw域其他地址到ww域的telnet功能。

其他几种静态转换配置

这里再补充几种相对来说不常用的静态转换配置。

入方向一对一

关键命令：nat static inbound global_ip local_ip，ip route-static local_ip global_ip/next_ip

入方向的静态地址转换通常用于与接口上的出方向动态地址转换（nat outbound）、内部服务器（nat server）或出方向静态地址转换（nat static outbound）配合以实现双向NAT，不建议单独配置。

对于入方向静态地址转换，需要手动添加路由：目的地址为静态地址转换配置中指定的local-ip或local-network；下一跳为静态地址转换配置中指定的外网地址，或者报文出接口的实际下一跳地址。

入方向一对一静态地址转换用于实现一个内部私有网络地址与一个外部公有网络地址之间的转换，具体过程如下：

  • 对于经过该接口发送的内网访问外网的报文，将其目的IP地址与指定的内网IP地址local-ip进行匹配，并将匹配的目的IP地址转换为global-ip。

  • 对于该接口接收到的外网访问内网的报文，将其源IP地址与指定的外网IP地址global-ip进行匹配，并将匹配的源IP地址转换为local-ip。

如果接口上配置的静态地址转换映射中指定了acl参数，则仅对符合指定ACL permit规则的报文进行地址转换。

上面是摘抄的H3C官方对于入方向静态转换的一些描述。换个思路，外网有个IP地址需要访问内网，本来可以在外网侧配置一个出方向的一对一静态转换(源NAT)，但是由于外网侧的设备不具备nat功能，所以在我们内网侧配置入方向一对一静态转换（反向snat？）。

重点是这里的global-ip是指的外网想要访问内网的那个外网IP地址，local-ip指的是我们配置的那个nat地址。

比如模拟环境中ww域的ww_web需要通过nat访问nw域的nw1，nat地址为10.10.10.253，那么global-ip就是11.11.11.11，local-ip就是10.10.10.253。接下来实际配置并验证一下

1
2
3
4

# 配置入方向静态转换
[fw]nat static in 11.11.11.11 10.10.10.253
# 再手动为之添加静态路由
[fw]ip route-static 10.10.10.253 32 10.10.10.1

可以看到从ww_web发起telnet的时候，nw1显示的登录用户IP是10.10.101.253(nat地址)；而从nw1向nat地址发起telnet的时候，最终也成功登录到了ww_web。

出方向地址段

连续地址段

假设模拟环境中nw域中1.1.1.1-1.1.1.11都需要nat访问ww域，选择配置地址段出方向静态转换

1
2
3
4
5
6

# 先清除之前配置的一对一静态转换
[fw]undo nat static inbound 11.11.11.11 10.10.10.253
[fw]undo nat static outbound 1.1.1.11 10.10.10.254
[fw]undo ip route-static 10.10.10.253 32 10.10.10.1
# 配置地址段出方向静态nat
[fw]nat sta out net-to-net 1.1.1.1 1.1.1.11 global 10.10.10.16 28

关键命令就是nat sta out net-to-net [内网开始ip] [内网结束ip] global [nat-ip 掩码]。

上文中的配置中，1.1.1.1将被转换为10.10.10.16，1.1.1.2被转换为10.10.10.17，以此类推，1.1.1.11被转换为10.10.10.27。

不过在本次模拟中，如果nwhx向ww域发起telnet，不会被nat转换，因为他会以8.8.8.2为源IP发起请求。

地址集合

如果是一个不连续的地址合集需要nat访问ww域，那么可以先创建IPv4地址对象组，然后再在nat中引用。

假设模拟环境中有1.1.1.1、1.1.1.11-1.1.1.20、1.1.1.128-1.1.1.143需要nat访问ww域。

# 创建源头地址对象组
[fw]object-group ip address source_ip
[fw-obj-grp-ip-source_ip]security-zone nw
[fw-obj-grp-ip-source_ip]network host address 1.1.1.1
[fw-obj-grp-ip-source_ip]net range 1.1.1.11 1.1.1.20
[fw-obj-grp-ip-source_ip]net sub 1.1.1.128 28
[fw-obj-grp-ip-source_ip]qu
# 创建nat地址对象组
# 内部需要nat的ip总共有27个，这里创建nat地址组也至少需要27个
[fw]object-g ip add nat_ip
[fw-obj-grp-ip-nat_ip]sec ww
[fw-obj-grp-ip-nat_ip]net sub 10.10.10.96 28
[fw-obj-grp-ip-nat_ip]net host add 10.10.10.88
[fw-obj-grp-ip-nat_ip]net ran 10.10.10.128 10.10.10.138
# 配置nat
[fw]nat sta out obj source_ip obj nat_ip

关键命令是：nat sta out obj [源地址对象组] obj [nat地址对象组]。

经上述配置后，模拟器上从nw1(1.1.1.11)向ww_web(11.11.11.11)发起telnet测试，发现nat并没有生效。感觉可能是地址对象组配置有问题，再次细看一遍说明文档，发现官方提醒：基于地址对象组的出方向静态地址转换引用的IPv4地址对象组中，只能存在一个主机对象（host）或者一个子网对象（subnet），否则引用不生效。

然而经过各种测试，最后两边地址组只剩下net range 1.1.1.11 1.1.1.20、net range 10.10.10.11 10.10.10.20，但还是无法生效，或者我这个版本的HCL模拟器不支持？

大量用户nat访问外网

动态nat有好几种配置方式，其中一种简易配置是直接使用出口IP作为nat地址，这种nat方式又叫做Easy IP；另外根据nat过程中是否转换端口号又可以分为PAT、No-PAT。然后配置的时候，对于内网需要nat的源地址范围的确定方式，又有基于acl策略和基于IP地址对象组两种。

基于acl策略

Easy IP

这个最简单，就是把内网需要nat的IP地址，直接转换为外网接口地址。

假设要把1.1.1.0/28网段的内网用户nat转换后访问外网，nat地址为防火墙出口IP: 10.10.10.2。

1
2
3
4
5
6
7
8
9

# acl确认需要nat的内网用户
<fw>sys
[fw]acl basic 2001
[fw-acl-ipv4-basic-2001]description 1.1.1.0/28 nat to internet
[fw-acl-ipv4-basic-2001]rule permit source 1.1.1.0 0.0.0.7
[fw-acl-ipv4-basic-2001]qu
# 出口配置easy ip并启用nat统计
[fw]int g1/0/1
[fw-GigabitEthernet1/0/1]nat outbound 2001 counting

这样配置之后就算是成功了，配置命令最后的counting，是启用nat命中次数统计。

此时我们可以在pc_4主机上对ww域的发起ping测试，然后在防火墙上运行dis nat outbound，就会看到结果里的NAT counting: 0数据发生了变化，或者浏览器登录防火墙，在相关NAT策略中也能看到命中次数发生了变化。

使用nat地址组

如果我们的外网IP地址有富余，也可以配置nat地址组，让内网IP地址使用地址组中的IP作为访问外网的源IP。

这里我们假设使用10.10.10.5、10.10.10.6做为1.1.1.8/28网段访问外网的源IP

# 配置nat地址组
[fw]nat address-group 0
[fw-address-group-0]add 10.10.10.5 10.10.10.6
[fw-address-group-0]qu
# acl确认内网地址范围
[fw]acl ba 2002
[fw-acl-ipv4-basic-2002]des 1.1.1.8/28 nat
[fw-acl-ipv4-basic-2002]rule per sou 1.1.1.8 0.0.0.7
[fw-acl-ipv4-basic-2002]qu
# 出口配置nat
[fw]int g1/0/1
[fw-GigabitEthernet1/0/1]nat out 2002 add 0 counting

基于IP地址对象组

以上是基于acl策略确认需要nat的地址范围来配置的nat，同时，也可以使用IP地址对象组来确认需要nat的地址范围。

Easy IP

还是上面的例子，假设要把1.1.1.0/28网段的内网用户nat转换后访问外网，nat地址为防火墙出口IP: 10.10.10.2。

[fw]object-group ip add nw_a
[fw-obj-grp-ip-nw_a]des nw a IP
[fw-obj-grp-ip-nw_a]sec nw
[fw-obj-grp-ip-nw_a]net sub 1.1.1.0 28
# 配置地址对象组，所属安全域和IP地址范围
[fw]object-group ip add nw_all
[fw-obj-grp-ip-nw_all]des nw all IP
[fw-obj-grp-ip-nw_all]sec nw
[fw-obj-grp-ip-nw_all]net sub 1.1.1.0 24
[fw-obj-grp-ip-nw_all]qu
# 配置nat策略
[fw]nat policy 
[fw-nat-policy]rule name nw_nat
[fw-nat-policy-rule-nw_nat]des all nw ip nat
[fw-nat-policy-rule-nw_nat]source-ip nw_all
[fw-nat-policy-rule-nw_nat]outbound-interface g1/0/1
[fw-nat-policy-rule-nw_nat]action easy-ip
[fw-nat-policy-rule-nw_nat]qu
[fw-nat-policy]qu

[fw-address-group-1]add 10.10.10.10 10.10.10.13
[fw-address-group-1]qu
# 修改nat策略
[fw]nat po
[fw-nat-policy]rule name nw_nat
[fw-nat-policy-rule-nw_nat]action address-group 1 no-pat

此时我们再telnet测试，就会发现源IP被转换成了nat地址组中的某个IP，而且端口也不会变化。

入方向动态转换

跟静态转换类似，动态nat也可以配置入方向的nat，将大量外网IP地址转换为少量的一个或几个nat地址组，这里省略…

限制端口范围(nat444)

在上文基于acl策略的No-PAT中简单提了几嘴，在配置动态转换的时候，如果不明确指定，在转换地址的时候，默认使用PAT模式，会随机转换端口，并且同一个IP在下一次nat的时候，转换的端口可能也会不一样。

其实还可以通过配置nat444地址转换，实现基于端口块的公网IP地址复用，使一个私网IP地址在一个时间段内独占一个公网IP地址的某个端口块。

还是以nw域1.1.1.0/24网段nat访问ww域为例，nat444地址转换有两种配置方式

动态nat444在配置上，和基于acl策略且使用nat地址组的普通动态转换很像，只是对于nat地址组额外配置了一个可用的nat端口块范围。

[fw]acl ba 2000
[fw-acl-ipv4-basic-2000]rule 0 permit source 1.1.1.0 0.0.0.255
[fw-acl-ipv4-basic-2000]qu
# 配置nat地址及端口范围
[fw]nat address-group 10
[fw-address-group-10]address 10.10.10.2 10.10.10.2
[fw-address-group-10]port-range 62500 65535
[fw-address-group-10]port-block block-size 10 extended-block-number 1
[fw-address-group-10]qu
[fw]int g1/0/1
# 取消刚才的静态nat444
[fw-GigabitEthernet1/0/1]undo nat outbound port-block-group 1
# 配置动态nat444
[fw-GigabitEthernet1/0/1]nat outbound 2000 address-group 10 counting

如同刚才的静态nat444，经过几次telnet测试，感觉就是这个动态nat444相对于静态nat444，端口释放更快，telnet连接断开之后，端口好像也同步释放了，下一次telnet别的IP时，端口号没有发生变化。
